<objective>
Close the Phase 3 verification gap for editable WAV output presets.

Purpose: The conversion pipeline already honors `AppSettings.audioPreset`, but users cannot currently change that preset from the production converter UI. This plan adds the missing edit path so `CONV-03` is true end to end: the user can choose sample rate, bit depth, and channel handling; the app persists those choices; the preset strip reflects them; and conversion requests use the selected values.

Output: `AudioConverterViewModel` preset state/update API, compact SwiftUI preset controls in `AudioConverterView`, and focused tests proving persistence plus conversion handoff.
</objective>

<threat_model>
Trust boundaries:
- User-edited preset values cross from SwiftUI controls into persisted app settings.
- Persisted preset values later control file naming and WAV conversion output parameters.
- Invalid or unsupported preset values could produce surprising output, failed conversions, or misleading UI.

STRIDE register:
- threat_id: T-03-PRESET-TAMPER
  category: Tampering
  component: AudioConverterViewModel.updateWAVPreset
  disposition: mitigate
  severity: medium
  mitigation_plan: Restrict UI choices to explicit supported values: sample rates `44100`, `48000`, `88200`, `96000`; bit depths `16`, `24`, `32`; channel modes `.preserveMonoStereo`, `.mono`, `.stereo`. Persist only those values.
- threat_id: T-03-PRESET-MISLEAD
  category: Repudiation
  component: AudioConverterView.presetStrip
  disposition: mitigate
  severity: medium
  mitigation_plan: Render `viewModel.presetSummaryText` from the current `AudioPreset`, not a static literal, so the visible target spec matches the settings used by conversion.
- threat_id: T-03-PRESET-OUTPUT
  category: Tampering
  component: AudioConverterViewModel queued rows
  disposition: mitigate
  severity: medium
  mitigation_plan: Refresh queued convertible row planned output names after preset edits so filenames and conversion requests both reflect the selected settings.
</threat_model>
