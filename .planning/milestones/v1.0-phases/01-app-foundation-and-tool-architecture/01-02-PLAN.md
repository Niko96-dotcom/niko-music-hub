<objective>
Harden the feature registry and tool context boundary that future tools will use.

Purpose: The app foundation only works if new tools can be added without editing existing feature internals or stuffing logic into a giant shell model. This plan turns the minimal scaffold contracts into tested AppCore APIs for feature metadata, view creation, capabilities, registry validation, and shared service injection.

Output: Immutable feature registry, clean ToolFeature protocol with view factory, ToolContext service protocols, duplicate-id and isolated-feature tests, and updated Dev Tool wiring through the refined boundary.
</objective>

<threat_model>
Trust boundaries:
- AppComposition is trusted startup code that chooses which tools are registered.
- ToolFeature implementations are internal modules that receive shared services only through ToolContext.
- AppCore tests are the guardrail against duplicate ids, hidden globals, and future shell coupling.

STRIDE register:
- threat_id: T-01-REG
  category: Tampering
  component: ToolRegistry
  disposition: mitigate
  severity: medium
  mitigation_plan: Reject duplicate ToolFeatureID values deterministically and test the thrown error.
- threat_id: T-01-GLOBAL
  category: Elevation of Privilege
  component: Feature registration
  disposition: mitigate
  severity: medium
  mitigation_plan: Keep registry immutable and prohibit automatic self-registration or global mutable registries in source.
- threat_id: T-01-CAP
  category: Spoofing
  component: ToolMetadata / ToolCapability
  disposition: mitigate
  severity: low
  mitigation_plan: Capability flags must be explicit metadata so the shell can distinguish file-producing and job-running tools.
</threat_model>
