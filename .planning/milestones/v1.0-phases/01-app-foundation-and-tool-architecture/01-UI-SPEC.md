# Phase 01 - UI Design Contract

> Visual and interaction contract for Phase 1: App Foundation and Tool Architecture. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

Phase 1 creates the first visible native macOS shell for Outside Cubase Hub. The UI must prove the app can list registered tools, open directly into the first registered tool, keep the active tool primary, expose the output inbox without letting it dominate, and make shared settings/job primitives visible enough to validate the foundation.

Do not show disabled roadmap placeholders for BPM, conversion, recording, or downloader tools in Phase 1. The only tool shown should be a real registered dummy/dev tool created through the same feature boundary future tools will use.

## Design System

| Property | Value |
|----------|-------|
| Tool | Manual native macOS SwiftUI contract |
| Preset | Not applicable |
| Component library | SwiftUI system controls with AppKit interop for Finder/file actions |
| Icon library | SF Symbols |
| Font | macOS system font, SF Pro Text/Display via SwiftUI `.system` |

### Component Contract

| Component | Contract |
|-----------|----------|
| App shell | `NavigationSplitView` or equivalent native split layout with sidebar primary navigation and active tool detail view. |
| Sidebar tool list | Registered tools only, ordered by `ToolRegistry`; row icon, short label, and active selection state. |
| Active tool view | Main visual focus. In Phase 1, the dummy/dev tool must make registration and shared context visible without becoming a fake product feature. |
| Output inbox | Always reachable as a trailing inspector, drawer, split column, or toolbar-revealed panel. It is secondary to the active tool. |
| Settings access | Native toolbar/menu entry for output folder and stored defaults. Avoid a large first-run wizard. |
| Job surface | Compact list or status strip showing queued, running, completed, failed, and canceled states when sample/dev jobs exist. |
| File actions | Use native affordances for Choose Folder, Reveal in Finder, and drag-ready file rows when outputs exist. |

## Layout And Interaction

| Area | Contract |
|------|----------|
| Window | Minimum usable size: 900px wide by 600px tall. Content must remain legible at minimum size. |
| Sidebar | Width range 220-280px. Use a compact list, not card navigation. |
| Detail area | Minimum 540px wide. The active registered tool owns this space. |
| Output inbox | Width range 280-340px when shown as an inspector/split column; collapsible if the window is narrow. |
| Launch state | Select the first registered tool automatically. No welcome dashboard. |
| Navigation | Sidebar selection changes active tool. Output inbox access must not change selected tool. |
| Keyboard | Support native focus order; Command-comma should open settings when settings exist. |

### Visual Hierarchy

The primary focal point is the active tool header and its first relevant action. The sidebar is the orientation layer, the active tool is the work layer, and the output inbox is the handoff layer. The eye should move in this order:

1. Active tool title and status.
2. Primary action in the active tool or settings surface.
3. Recent job/output status.
4. Sidebar navigation.
5. Output inbox details.

Icon-only controls must have accessibility labels and tooltips. Use text labels for primary commands; reserve icon-only buttons for familiar native actions such as Reveal in Finder, collapse inspector, or clear/cancel when the label would create clutter.

## Spacing Scale

Declared layout values. All spacing tokens are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline metadata separators, tight label/icon spacing |
| sm | 8px | Compact control gaps, sidebar row inner spacing |
| md | 16px | Default element spacing, panel padding, grouped settings rows |
| lg | 24px | Tool header spacing, inspector section padding |
| xl | 32px | Major content gaps inside the detail area |
| 2xl | 48px | Empty state vertical rhythm and large tool sections |
| 3xl | 64px | Rare page-level spacing for wide window breathing room |

Exceptions: none for layout spacing. Native macOS controls may keep platform intrinsic heights, but custom padding and gaps must use the tokens above.

## Typography

Use exactly four text sizes and exactly two weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Caption / metadata | 12px | Regular 400 | 1.35 |
| Body | 13px | Regular 400 | 1.45 |
| Section heading | 16px | Semibold 600 | 1.25 |
| Screen title | 22px | Semibold 600 | 1.20 |

Letter spacing: 0. Do not scale font sizes with viewport width. Use truncation only for file paths, URLs, or long helper-tool paths; user-facing labels and job messages should wrap cleanly.

## Color

Use native semantic colors where possible, backed by the fixed reference values below for planning and review. The palette should feel calm and local-machine native, with a small amount of production-tool character.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F7F8FA | Window background and main detail surfaces |
| Secondary (30%) | #E7EAEE | Sidebar, output inbox, grouped settings surfaces, dividers |
| Accent (10%) | #0B6E69 | Primary CTA, active sidebar indicator, keyboard focus ring, running progress fill |
| Destructive | #C7362F | Cancel job, remove output metadata, destructive confirmation emphasis only |

Accent reserved for: primary CTA, active sidebar indicator, keyboard focus ring, running progress fill. Do not use the accent for every button, every icon, or decorative backgrounds.

State colors:

| State | Value | Usage |
|-------|-------|-------|
| Completed | #2F7D32 | Completed job/output status only |
| Failed | #C7362F | Failed job/status text and destructive emphasis |
| Canceled | #6B7280 | Canceled job state and inactive metadata |
| Queued | #4B5563 | Queued job state |

## Copywriting Contract

Use terse, concrete labels. The app should sound like a focused studio utility, not onboarding software.

| Element | Copy |
|---------|------|
| Primary CTA | Choose Output Folder |
| Secondary CTA | Run Sample Job |
| Sidebar item for dummy/dev tool | Dev Tool |
| Output inbox label | Output Inbox |
| Empty state heading | No outputs saved yet |
| Empty state body | Files created by registered tools will appear here for reveal and drag-out. Choose an output folder before running file-producing tools. |
| Error state | Could not open the output folder. Choose a local folder you can write to, then try again. |
| Missing folder state | Output folder is unavailable. Choose Output Folder to reconnect a local destination. |
| Destructive confirmation | Cancel Job: Stop this job? Progress will be discarded, but created files stay in the output folder. |

Avoid generic one-word or vague command labels. Use action-specific labels such as Choose Output Folder, Reveal in Finder, Stop Job, or Remove Output Record.

## Empty, Loading, And Error States

| State | Contract |
|-------|----------|
| No registered tools | Blocked app configuration state. Show "No tools registered" with guidance for the developer to register a `ToolFeature` in the composition root. This should not occur in normal Phase 1 manual verification. |
| No outputs | Show the output inbox empty state copy from the copywriting contract. Keep the active tool visible. |
| No jobs | Hide the job list or show one compact line: "No jobs running." Do not dedicate a large panel to an empty queue. |
| Running job | Show state, progress if known, and latest message. Include Stop Job only when cancellation is supported. |
| Failed job | Show concise problem text plus the next recovery action. Preserve details in logs/diagnostics, not the primary row. |
| Missing output file | Mark the output item as missing and offer Reveal in Finder only when the parent folder still exists. |

## Accessibility And Native Behavior

| Requirement | Contract |
|-------------|----------|
| VoiceOver | Every sidebar item, icon-only action, job state, and file action must have an accessibility label. |
| Focus | Keyboard focus order follows sidebar, active tool, output inbox. Focus rings use the accent color only. |
| Contrast | Text and status colors must meet WCAG AA contrast against their surfaces. |
| Motion | Avoid custom animation except native split-view transitions and subtle progress changes. |
| Native menus | Settings should be reachable through Command-comma when implemented. Finder reveal should use native AppKit behavior. |
