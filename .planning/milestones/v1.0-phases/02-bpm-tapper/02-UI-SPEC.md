# Phase 02 - UI Design Contract

> Visual and interaction contract for Phase 2: BPM Tapper. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

Phase 2 adds the first real production utility to Outside Cubase Hub: a native offline BPM tapper that replaces opening an external website during a Cubase session. The UI must feel fast, calm, and keyboard-friendly. It must support a focused tap zone, spacebar tapping while the BPM tool is active, reset by button and Escape, half-time and double-time adjustment modes, plain-number clipboard copy, one-click saving to local history, and a recent saved-history list.

Do not add audio-file BPM analysis, global shortcuts outside the active app, Cubase project integration, naming templates, note-taking, or sample-prep workflows in this phase.

## Design System

| Property | Value |
|----------|-------|
| Tool | Manual native macOS SwiftUI contract |
| Preset | Not applicable |
| Component library | SwiftUI system controls with AppKit interop only where needed for clipboard behavior |
| Icon library | SF Symbols |
| Font | macOS system font, SF Pro Text/Display via SwiftUI `.system` |

### Component Contract

| Component | Contract |
|-----------|----------|
| BPM tool registration | Register as a normal `ToolFeature`; sidebar short label is `BPM Tapper` with an SF Symbol that reads as tempo/tapping, such as `metronome` if available on the target SDK or `waveform` as fallback. |
| Tool header | Compact title row inside the active tool detail area. Show `BPM Tapper` and one short status line. Do not create a landing page or separate dashboard. |
| Tap surface | One large focused control, visually dominant, clickable only inside its bounds. Stable height from idle through active states. It must accept focus so Space taps are scoped to this tool. |
| BPM readout | Large plain-number tempo value with a small `BPM` label. Show the adjusted value as primary and preserve original tapped BPM as secondary context when half-time or double-time is active. |
| Tap progress metadata | Compact line showing tap count and estimator confidence, such as `2 taps - first estimate` or `5 taps - stable average`. |
| Adjustment modes | Native segmented control with `Original`, `Half-Time`, and `Double-Time`. Changing mode updates displayed, copied, and saved BPM while preserving original tapped context. |
| Primary actions | Text buttons for `Copy BPM`, `Save BPM`, and `Reset Taps`. Use bordered prominent styling only for the next most useful action after a BPM exists. |
| Inline confirmations | Small inline confirmation text near the triggering action for copy and save. Do not use system notifications for Phase 2 confirmations. |
| Recent history | Compact list in the main BPM view showing recent saved BPM values, timestamp, adjustment context, and a row-level `Copy Saved BPM` action. |
| Clear history | Secondary destructive action below the recent list. Confirm before clearing all saved BPM history. No individual row deletion in Phase 2. |
| Output inbox | Remains visible as the shared inspector from the shell. BPM history is not a file-producing output inbox workflow. |

## Layout And Interaction

| Area | Contract |
|------|----------|
| Detail area | Keep Phase 1 shell constraints: active tool minimum width 540px inside the split view. BPM content should fit cleanly at the 900px minimum window width. |
| Content width | Constrain BPM controls to an ideal width of 440-560px and align them top-leading. Avoid stretching the tap target to full ultra-wide desktop width. |
| Tap surface | Minimum size 320px wide by 180px tall; ideal size about 440px by 220px. Use a stable aspect/height so tap state, labels, and BPM changes do not shift surrounding content. |
| Result cluster | Keep BPM number, unit, adjustment context, and tap metadata visually grouped directly above or inside the tap surface. |
| Action row | Place `Copy BPM`, `Save BPM`, and `Reset Taps` close to the readout. Keep actions enabled only when they have a valid target, except `Reset Taps` may remain available once a run starts. |
| History list | Show below the tap workflow, not in the output inbox. Keep rows dense enough for studio use: BPM first, then timestamp/context, then row copy action. |
| Empty history | Use a compact empty state in the history area; do not give empty history equal visual weight to the tap surface. |
| Keyboard | Spacebar records a tap only when the BPM tool is active/focused. Escape resets the current tap run. Native tab order must reach tap surface, actions, adjustment control, history actions, and clear history. |
| Long pause | The first tap after the timeout starts a fresh run and updates status without an alert. |
| Outlier tap | Ignore obvious outlier intervals for the estimate, keep the run alive, and show a short inline status. |

### Visual Hierarchy

The primary focal point is the BPM readout and large tap surface. The adjustment mode is the second focal point once a result exists, followed by copy/save/reset actions and then recent history. The output inbox inspector remains visually secondary and must not compete with the tempo surface.

The eye should move in this order:

1. BPM number or idle `Tap Tempo` call to action.
2. Tap surface state and tap progress metadata.
3. Adjustment segmented control.
4. Copy, Save, and Reset actions.
5. Recent saved tempos.
6. Shared output inbox inspector.

Icon-only controls must have accessibility labels and tooltips. Use text labels for BPM commands; reserve icons for familiar small affordances only if the text label would create clutter.

## Spacing Scale

Declared layout values. All custom spacing tokens are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Unit label gaps, inline metadata separators, icon/text gaps |
| sm | 8px | Control gaps, compact history-row spacing, confirmation spacing |
| md | 16px | Default group spacing, action-row gaps, section inner padding |
| lg | 24px | Tool header spacing, tap workflow section padding |
| xl | 32px | Gap between tap workflow and history |
| 2xl | 48px | Empty-state vertical rhythm when history is empty |
| 3xl | 64px | Rare wide-window breathing room only |

Exceptions: none for custom layout spacing. Native SwiftUI controls may keep platform intrinsic heights, but custom padding and gaps must use the tokens above.

## Typography

Use exactly four text sizes and exactly two weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Caption / metadata | 12px | Regular 400 | 1.35 |
| Body | 13px | Regular 400 | 1.45 |
| Section heading / tool title | 16px | Semibold 600 | 1.25 |
| BPM display | 40px | Semibold 600 | 1.10 |

Letter spacing: 0. Do not scale font sizes with viewport width. BPM values must keep a stable line box across idle, estimating, and adjusted states. Long timestamps or storage errors should wrap or truncate in metadata positions only.

## Color

Reuse the Phase 1 native palette so the BPM tool feels like part of the same production bench. Prefer native semantic colors in SwiftUI, backed by these fixed reference values for planning and review.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F7F8FA | Window background and main BPM detail surface |
| Secondary (30%) | #E7EAEE | Sidebar, output inbox, tap surface fill, history rows, dividers |
| Accent (10%) | #0B6E69 | Active tap-surface border, primary BPM action, selected adjustment segment emphasis, keyboard focus ring |
| Destructive | #C7362F | Clear History confirmation, destructive emphasis only |

Accent reserved for: active tap-surface border, primary BPM action after a valid estimate exists, selected adjustment segment emphasis, keyboard focus ring. Do not use the accent for every button, every history row, every icon, or decorative backgrounds.

State colors:

| State | Value | Usage |
|-------|-------|-------|
| Saved / copied | #2F7D32 | Inline success confirmation text only |
| Ignored tap / warning | #8A5A00 | Outlier or stale-pause inline status only |
| Error | #C7362F | History persistence failure and destructive confirmation emphasis |
| Muted metadata | #6B7280 | Tap count, timestamps, original BPM context, inactive states |

## Copywriting Contract

Use terse, concrete labels. The tool should sound like a studio utility that is already ready to work.

| Element | Copy |
|---------|------|
| Primary CTA | Tap Tempo |
| Copy action | Copy BPM |
| Save action | Save BPM |
| Reset action | Reset Taps |
| Adjustment labels | Original, Half-Time, Double-Time |
| Row copy action | Copy Saved BPM |
| Clear history action | Clear History |
| Empty state heading | No tempos saved yet |
| Empty state body | Saved BPM results will appear here with their time and adjustment mode. Tap a tempo, then Save BPM to keep it for this session. |
| Idle tap state | Tap the pad or press Space |
| First estimate state | First estimate ready. Keep tapping to steady it. |
| Stable estimate state | Stable average from recent taps. |
| Long pause state | New tap run started. |
| Outlier state | Ignored one uneven tap. Keep tapping. |
| Copy confirmation | BPM copied |
| Save confirmation | BPM saved |
| Error state | Could not save this BPM. Check local app storage, then try Save BPM again. |
| Destructive confirmation | Clear History: Clear all saved tempos? This keeps the current tap run but removes saved BPM history. |

Avoid generic one-word command labels such as `Save`, `Copy`, `Reset`, `OK`, or `Cancel` in the BPM surface. Use action-specific labels from the table.

## Empty, Loading, And Error States

| State | Contract |
|-------|----------|
| No taps | Show the idle tap state, disabled copy/save actions, adjustment mode on Original, and empty or existing history below. |
| One tap | Show tap count and prompt for one more tap. Do not display a fake BPM estimate. |
| Two taps | Show the first estimate and the first estimate state. Enable adjustment, copy, and save. |
| Stable run | Show stable estimate state once enough intervals are available. Continue updating from recent intervals without layout shift. |
| Long pause | Start a fresh run on the next tap and show long pause state inline. Do not show a modal or error. |
| Outlier ignored | Keep the previous estimate visible and show outlier state inline for a short time. |
| Copy succeeded | Show copy confirmation near `Copy BPM` or the row copy action. Plain clipboard value only, such as `128`. |
| Save succeeded | Append the current displayed result to history and show save confirmation near `Save BPM`. |
| Save failed | Keep the current run intact. Show the error-state copy near the action and log details through diagnostics if available. |
| No history | Show the empty-state copy in the history area. Keep the tap workflow dominant. |
| Clear history confirmed | Remove all saved history rows and return the history area to the empty state. |

## Accessibility And Native Behavior

| Requirement | Contract |
|-------------|----------|
| VoiceOver | Tap surface, BPM readout, adjustment control, copy/save/reset, row copy actions, and clear history must have accessibility labels. The BPM readout should announce value and adjustment mode. |
| Focus | Initial focus should land on the tap surface when the BPM tool opens. Focus rings use the accent color only. |
| Keyboard | Spacebar and Escape behavior must be discoverable through native menu/tooltip/accessibility text, not visible instructional panels. |
| Hit targets | Tap surface is intentionally large. Other custom controls should preserve native macOS target sizes and spacing. |
| Contrast | Text, status colors, accent border, and destructive color must meet WCAG AA contrast against their surfaces. |
| Motion | Use only subtle native-state changes, such as a brief tap-surface border flash or confirmation fade. Avoid rhythmic pulsing or decorative animation while the user is tapping. |
| Clipboard | Copy writes only the displayed plain number. It must not include `BPM`, mode text, timestamp, or formatting. |
