<name>Task 1: Implement yt-dlp progress parsing and stderr streaming</name>
<files>Sources/FeatureDownloader/YtDlpDownloader.swift, Tests/FeatureDownloaderTests/DownloadProgressParsingTests.swift</files>
<read_first>Sources/FeatureDownloader/YtDlpDownloader.swift, Sources/AppCore/Services/ExternalProcessRunning.swift, .planning/phases/05-downloader-hub/05-CONTEXT.md</read_first>
<action>Enhance `YtDlpDownloader.download()` to stream progress:

1. Change the runner to support async stderr streaming (needed for real-time progress). Instead of using the synchronous `FoundationExternalProcessRunner`, create a live streaming variant that:
   - Runs the process asynchronously
   - Reads stderr line-by-line as data arrives on `errorPipe.fileHandleForReading`
   - Calls `progressHandler` with each full line
   - Extracts progress percentage from yt-dlp progress lines (format: `[download]  X%` or `[download]  X% of ...`)
   - Feeds percentage to `JobProgress.update(progress:)` and raw lines to `JobProgress.log()`
   - Collects final stdout/stderr after process terminates

2. Parse progress from yt-dlp stderr lines:
   - `%` character indicates a progress line — extract the percentage number before `%`
   - Lines with `[download]` prefix are progress lines
   - Lines with `[ExtractAudio]` or `[info]` are info/log lines
   - Everything else is raw log output

3. For output file tracking: yt-dlp prints final output filename to stderr when complete. Look for pattern `[download] Destination: ...` to capture the final output URL.

4. The async stderr streaming is essential — progress updates must appear in real-time during download, not after completion.

Note: You may need to create an `AsyncExternalProcessRunner` variant that wraps the synchronous one but adds async stderr reading. Follow the same `ExternalProcessRunning` port contract.
</action>

<name>Task 2: Implement retry with exponential backoff and partial file cleanup</name>
<files>Sources/FeatureDownloader/DownloaderUseCase.swift</files>
<read_first>Sources/FeatureDownloader/DownloaderUseCase.swift, Sources/FeatureDownloader/YtDlpDownloader.swift, Sources/FeatureDownloader/DownloaderViewModel.swift, .planning/phases/05-downloader-hub/05-CONTEXT.md</read_first>
<action>Enhance `DownloaderUseCase` retry and cleanup logic:

1. `DownloadJobOptions` already has `retries: Int = 3`. In the enqueued job operation:
   - Wrap the download call in a retry loop
   - For transient errors (network timeout, server error 5xx, connection reset), retry up to `options.retries` times
   - Exponential backoff: 2s, 4s, 8s between retries
   - Non-transient errors (404, forbidden, unsupported site, yt-dlp error) do NOT retry
   - Track retry count and backoff timing

2. Transient error detection: check yt-dlp stderr for patterns:
   - `HTTP Error 5xx`, `Connection reset`, `Connection timed out`, `Temporary failure`
   - Exit codes 1,2,4 with transient-looking stderr

3. On final failure after all retries:
   - Identify any partial output files created during the failed download
   - Delete those partial files so no orphan files remain in the output folder
   - Mark job as failed with a message that distinguishes retryable vs permanent failures

4. For playlists (multiple files): track each output file path as yt-dlp reports them, add each to inbox separately with the same `dlSourceURL` metadata.
</action>
